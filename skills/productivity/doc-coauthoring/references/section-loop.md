# Stage 2 — Refinement and Structure

Goal: build the document one section at a time, through brainstorming, curation, and surgical edits.

## Agreeing the Structure

If the user knows the sections, ask which one to start with. If they don't, propose 3-5 sections appropriate to the doc type and template, and ask whether the shape works.

Order by uncertainty: start with the section carrying the most unknowns — usually the core proposal in a decision doc, the technical approach in a spec. Summaries and abstracts go last, once there is something to summarize.

## Scaffold

Write the file with every section header and a placeholder body (`[To be written]`). Both sides now have the same map. Tell the user where the file is.

## The Per-Section Loop

Announce the section, then run all six steps before moving on.

### 1. Clarify

Ask 5-10 specific questions about what this section must cover, derived from Stage 1 context and the section's purpose. Shorthand answers are fine.

### 2. Brainstorm

List 5-20 numbered candidate points, scaled to the section's complexity. Include:

- Context the user shared earlier and may have forgotten
- Angles and objections they have not raised

Offer to generate more if the list feels thin.

### 3. Curate

Ask which points to keep, cut, or combine, and ask for brief reasons — the reasons teach priorities for every later section. Model the format:

- `Keep 1,4,7,9`
- `Remove 3 (duplicates 1)`
- `Remove 6 (audience already knows this)`
- `Combine 11 and 12`

If they answer freeform instead ("looks good, but drop the migration stuff"), extract the selection and proceed rather than insisting on numbers.

### 4. Gap Check

Ask what is missing from the selection for this section specifically.

### 5. Draft

Replace the section's placeholder with real content via a targeted edit. Leave the rest of the file untouched.

On the first section only, set the working agreement: ask the user to say what to change rather than editing the file directly, because their change requests teach style for the following sections. If they edit anyway, keep the edit and treat it as a preference signal.

### 6. Refine

Apply each change request as a targeted edit. Confirm the edit; never reprint the document. Iterate until the user is satisfied.

After three consecutive rounds with no substantial change, ask what can be removed without losing information — plateau usually means padding, not perfection.

## Whole-Document Pass

At roughly 80% of sections complete, read the full document and report on:

- Flow and consistency across sections
- Redundancy and contradictions
- Generic filler that survived because cutting felt risky
- Whether every sentence carries weight

When all sections are drafted and refined, read it once more end to end, give final suggestions, and ask whether to move to reader testing.
